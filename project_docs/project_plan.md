# 气候政策多利益相关方博弈沙盘项目规划

## 1. 项目定位

- 项目名称：`全球气候政策多利益相关方博弈沙盘`
- 当前阶段目标：先做出一个`可运行、可解释、可展示`的 `1.0` 原型，用于课堂演示和后续迭代。
- 项目形态：一个基于 `Python` 的轻量级多智能体政策推演工具，后续可扩展为 `Streamlit Web App`。
- 课程对应关系：
  - 回应气候治理中的真实多方博弈问题
  - 体现 AI 在角色扮演、结构化输出和决策支持中的作用
  - 保留 Vibe Coding 和工具协同的课程风格

## 2. 核心科学问题

一句话版本：

> 如何构建一个结合政策文本、国家发展条件与气候脆弱性信息的多智能体推演系统，用于模拟气候政策在不同利益相关方之间的博弈过程，并识别其阻力来源与可能的妥协路径？

进一步拆解为三个子问题：

1. 不同利益相关方为什么会对同一项气候政策作出截然不同的回应？
2. 哪些条件会导致一项政策从“强烈反对”转向“条件性支持”？
3. AI 能否把多方对话沉淀成一份结构化、可解释的政策可行性报告？

## 3. 第一阶段交付目标

本阶段不追求做成完整研究工具，而是先完成一个最小可行闭环。

### 本阶段必须完成

- 一份清晰的项目规划文档
- 一个 `1.0` 可运行原型
- 一组默认角色与样例政策输入
- 一次完整的样例推演结果
- 一份适合课堂汇报的 `pipeline & AI 使用` 简报材料
- 完整留痕：代码、输入、输出、日志、说明文档

### 本阶段暂不强求

- 接入真实在线大模型 API
- 完整前端可视化界面
- 大规模真实数据清洗
- 严格的量化评估实验

## 4. 1.0 原型范围

### 输入

- 一条政策提案
- 一组政策属性
  - 减排强度
  - 碳关税强度
  - 气候资金支持
  - 技术转移支持
  - 过渡期灵活性
  - 是否允许 CCS 过渡技术

### 参与角色

- 发展中国家代表
- 发达国家联盟代表
- 传统能源/重工业代表
- 小岛国/环保 NGO 代表
- 观察员/秘书长

### 推演过程

- 固定 3 轮博弈
- 每轮输出各角色发言
- 根据博弈结果自动修改政策属性
- 由观察员在最后输出结构化报告

### 输出

- 逐轮对话记录
- 各角色最终立场
- 阻力来源总结
- 潜在妥协条件
- 简化评分
  - 落地可行性
  - 公平性
  - 阻力强度

## 5. 系统 Pipeline

```text
政策输入
  -> 政策解析
  -> 角色初始化
  -> 第 1 轮立场表达
  -> 观察员总结冲突点
  -> 自动调整政策条件
  -> 第 2 轮再博弈
  -> 再次总结并调整
  -> 第 3 轮形成相对稳定态度
  -> 输出结构化政策可行性报告
```

## 6. AI 在项目中的使用方式

本项目会分两层使用 AI。

### 第一层：当前 1.0 原型

- 采用`规则引擎 + Prompt 风格模板`的方式先跑通系统闭环
- 优点：
  - 不依赖 API，便于本地快速验证
  - 行为更稳定，便于解释和调试
  - 适合课堂展示“pipeline 和方法设计”

### 第二层：后续增强版

- 接入云端大模型 API
- 让 AI 负责：
  - 更自然的角色发言生成
  - 根据角色背景生成更丰富的论证
  - 输出更像真实政策简报的总结报告
- 保留规则层作为约束，避免“AI 自由发挥过头”

## 7. 数据策略

目前数据分三层，但 `1.0` 只做轻量映射，不做完整数据接入。

### 数据层一：政策文本

- 代表什么：国家公开承诺了什么
- 未来来源：`UNFCCC NDC Registry`
- 当前用途：为角色立场和争议议题提供真实政策背景

### 数据层二：国家指标

- 代表什么：排放、能源、发展条件
- 未来来源：`World Bank`
- 当前用途：为角色偏好和底线设计提供现实依据

### 数据层三：脆弱性与适应能力

- 代表什么：国家面临风险的程度与应对能力
- 未来来源：`ND-GAIN`
- 当前用途：强化“小岛国/高脆弱国家”的政策紧迫性

## 8. 技术路线

## 10. 风险与应对

### 风险一：项目看起来像普通聊天机器人

- 应对：强调结构化输入、规则状态机和报告输出

### 风险二：没有真实数据支撑

- 应对：明确三层数据路线，把 `1.0` 定位为方法原型

### 风险三：AI 发言不稳定

- 应对：先用规则引擎稳定跑通，再接 API 做增强

### 风险四：工程量失控

- 应对：第一版只保留 4 个核心角色、3 轮博弈、1 份报告

## 11. 当前判断

- 这个项目`完全适合做成课程原型`
- 当前最重要的不是一开始把系统做得很复杂，而是先形成：
  - 清晰的问题定义
  - 稳定的最小闭环
  - 可展示的样例输出
  - 明确的升级路径

## 12. 本轮开发任务

1. 搭建 `1.0` 原型代码
2. 跑通一条默认政策的完整推演
3. 保存样例结果为 `Markdown + JSON`
4. 整理经验、问题和下一步建议
5. 形成可直接用于课堂简单分享的材料
